# Backend-Legacy Agent Rules

## Critical Awareness

This package is the **TYPE AUTHORITY** for the entire Adaptic.ai platform. The Prisma schema in this package defines 62 models and 63 enums that serve as the canonical type definitions for the whole monorepo. Changes here propagate to:

- **Engine** (`@adaptic/engine`) -- direct build-time dependency. Imports types, enums, CRUD functions, and Apollo Client.
- **Utils** (`@adaptic/utils`) -- direct build-time dependency. Imports types and enums.
- **Platform** (`platform/`) -- runtime dependency via GraphQL and WebSocket. Consumes types through Apollo Client queries.

**Type resolution priority across the monorepo:**
`@adaptic/backend-legacy` -> `@adaptic/utils` -> `@adaptic/lumic-utils` -> `src/interfaces/`

## Before Making Changes

1. **Understand the FULL impact of schema changes.** Every model and enum change affects downstream packages.
2. **Check which packages consume the types you are modifying.** Search for usage in `engine/`, `utils/`, and `platform/` before changing any model.
3. **Plan migration strategy before touching the schema.** Non-trivial changes (field removal, type changes, renames) require a rollback plan.
4. **Run the full codegen pipeline after every change.** A partial rebuild will leave generated files out of sync.
5. **Verify dependent packages still build.** After changes, confirm `engine`, `utils`, and `platform` can still typecheck and build.

## Schema Modification Rules

1. **Never remove a field without checking all consumers.** Search the entire monorepo for references to the field before removing it. Downstream breakage may not be caught by this package's own build.
2. **Never rename a model without updating all imports.** Model names appear in import paths, GraphQL queries, selection sets, type strings, and consumer code across all packages.
3. **Always add new fields as optional or with defaults.** Required fields without defaults will break existing rows and may require data migrations. Use `?` for optional or `@default()` for defaulted fields.
4. **Run `npm run build` after changes to verify.** This executes the full 8-step pipeline: clean, generate, fix-imports, generate:selections, generate:functions, generate:strings, tsc, build:server.
5. **Run `npm run lint` and `npm run test` before committing.** Linting and tests catch issues the build alone may miss.
6. **Use Prisma schema comments (`///`) for documentation.** These propagate to generated TypeGraphQL decorators and GraphQL schema descriptions.

## Generated Code

- **NEVER manually edit generated files.** The following are ALL generated and overwritten on every build:
  - `src/generated/typegraphql-prisma/` (models, enums, resolvers, inputs, outputs)
  - `src/generated/selectionSets/` (62 model files + index.ts)
  - `src/generated/typeStrings/` (62 model files + index.ts)
  - `src/[A-Z]*.ts` (62 PascalCase model CRUD files)
  - `src/index.ts` (package entry point, regenerated by generate:functions)
- **If generated output is wrong, fix the generator input or config:**
  - Wrong types or models -> fix `prisma/schema.prisma`
  - Wrong import paths -> fix `fix-import-paths.cjs`
  - Wrong selection sets -> fix `src/modules/generateSelections.ts`
  - Wrong CRUD functions -> fix `src/modules/generator.ts` or `src/modules/parser.ts`
  - Wrong type strings -> fix `src/modules/generateStrings.ts`
- **Re-run codegen if generated files are out of sync.** Run `npm run build` (not just `npm run generate`) to ensure all 5 codegen steps execute.

## Migration Safety

1. **Never modify existing migration files.** There are 149 migrations in history. Altering them breaks the migration chain.
2. **Always test migrations on dev before production.** Use `npm run migrate:dev` for development, `npm run migrate` for production deploy.
3. **Include rollback strategy for non-trivial migrations.** Document how to reverse the migration if something goes wrong.
4. **Name migrations descriptively** when Prisma prompts for a name. Use clear descriptions like `added_fund_assignment_model` or `made_email_optional`.
5. **Be aware of data migrations.** Schema changes that alter column types or remove fields may require data migration scripts (see `src/scripts/` for examples).

## Type Export Rules

1. **Maintain backward compatibility when possible.** Adding fields is safe; removing or renaming fields is a breaking change.
2. **If breaking changes are needed, coordinate with engine and utils teams.** Both packages directly import from `@adaptic/backend-legacy`.
3. **Document breaking changes clearly.** Note what changed, why, and what consumers need to update.
4. **Verify the package entry point after codegen.** `src/index.ts` is regenerated by `generate:functions`. Ensure all expected exports are present.
5. **Test the published package shape.** The `files` field in `package.json` controls what ships to NPM: `dist/**/*.js`, `dist/**/*.d.ts`, `dist/**/*.js.map`.

These are generated. Any manual edits will be lost:

| Path                                  | Generated By                         |
| ------------------------------------- | ------------------------------------ |
| `src/generated/typegraphql-prisma/**` | `npm run generate` (prisma generate) |

## Verification Checklist

Before considering any change complete, verify:

- [ ] `npm run build` passes (all 8 pipeline steps)
- [ ] `npm run lint` passes
- [ ] `npm run test` passes
- [ ] Generated output counts are correct (62 selection sets, 62 CRUD files, 62 type strings)
- [ ] `src/index.ts` exports all expected models, types, enums, and functions
- [ ] No hand-edits in generated file paths
- [ ] Dependent packages (`engine`, `utils`) still typecheck after the change
